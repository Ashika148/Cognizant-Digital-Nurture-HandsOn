DIFFERENCE BETWEEN JPA, HIBERNATE AND SPRING DATA JPA


These three technologies are closely related but serve different purposes:

| Technology          | What it is                      | Role                                                   |
| ------------------- | ------------------------------- | ------------------------------------------------------ |
| **JPA**             | Specification (API/standard)    | Defines how Java objects should be mapped to databases |
| **Hibernate**       | Implementation (ORM framework)  | Implements JPA and provides additional features        |
| **Spring Data JPA** | Abstraction layer on top of JPA | Reduces boilerplate code for data access               |

### 1. JPA (Java Persistence API)

JPA is a **specification**, not a framework.

It defines:

* Annotations like `@Entity`, `@Id`, `@Column`
* Interfaces like `EntityManager`
* Rules for ORM (Object Relational Mapping)

Example:

```java
@Entity
public class Employee {

    @Id
    private Long id;

    private String name;
}
```

JPA itself cannot interact with the database. You need an implementation such as Hibernate.

Think of JPA as an **interface**:

```java
public interface Vehicle {
    void start();
}
```

It defines what should happen, not how.

---

### 2. Hibernate

Hibernate is the most popular **JPA implementation**.

It:

* Implements all JPA specifications
* Generates SQL
* Manages object-database mapping
* Provides extra features beyond JPA

Example:

```java
EntityManager em = entityManagerFactory.createEntityManager();

Employee emp = em.find(Employee.class, 1L);
```

Internally, Hibernate executes SQL like:

```sql
SELECT * FROM employee WHERE id = 1;
```

Hibernate-specific features:

* HQL (Hibernate Query Language)
* Second-level cache
* Batch processing
* Advanced fetching strategies

Without Spring Data JPA, you often write repositories manually:

```java
@Repository
public class EmployeeRepository {

    @PersistenceContext
    private EntityManager em;

    public Employee findById(Long id) {
        return em.find(Employee.class, id);
    }
}
```

---

### 3. Spring Data JPA

Spring Data JPA sits **on top of JPA/Hibernate**.

Its goal is to eliminate boilerplate repository code.

Instead of writing:

```java
@Repository
public class EmployeeRepository {
    @PersistenceContext
    private EntityManager em;

    public Employee findById(Long id) {
        return em.find(Employee.class, id);
    }
}
```

You simply write:

```java
public interface EmployeeRepository
        extends JpaRepository<Employee, Long> {
}
```

Spring automatically provides:

* `save()`
* `findById()`
* `findAll()`
* `deleteById()`

And even query generation:

```java
List<Employee> findByName(String name);

List<Employee> findByDepartmentAndSalaryGreaterThan(
        String department,
        Double salary);
```

Spring Data JPA generates the SQL automatically.

## Relationship Diagram

```text
Application
     |
     v
Spring Data JPA
     |
     v
JPA (Specification)
     |
     v
Hibernate (Implementation)
     |
     v
Database
```

In a typical Spring Boot application:

```xml
spring-boot-starter-data-jpa
```

includes:

* Spring Data JPA
* JPA API
* Hibernate (default implementation)

So when you call:

```java
employeeRepository.findById(1L);
```

The flow is:

```text
Spring Data JPA
       ↓
JPA API
       ↓
Hibernate
       ↓
MySQL/PostgreSQL
